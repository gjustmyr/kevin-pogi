FACULTY CLEARANCE STATUS - AUTO-CALCULATION WORKFLOW (March 6, 2026)
================================================================================

TERMINOLOGY CLARIFICATION:
--------------------------
⚠️ IMPORTANT: Different meanings:

A) "VALIDATED" = Individual files approved by dean (statistics count)
   - Shows as: "3 Validated", "15 Validated", etc.
   - Counts how many requirement files have been approved/checked by dean
   - Increments by 1 each time dean validates a file
   - Example: "15 Validated" means 15 out of 27 files are checked/approved

B) "CLEARED" = Overall faculty status (badge/status only)
   - Shows as: Green "CLEARED" badge
   - Only changes to "CLEARED" when ALL requirements are validated (27/27)
   - This is the faculty's overall clearance status, not a count

CLARIFICATION OF STATUS MEANINGS:
----------------------------------

1. REQUIREMENT SUBMISSION STATUS (Individual Files):
   - "pending": File submitted, awaiting dean review/validation
   - "validated": File approved/checked by dean ← THIS IS COUNTED IN STATISTICS
   - "returned": File rejected by dean, needs revision

2. FACULTY CLEARANCE STATUS (Overall Faculty Status - Shown as Badge):
   - "pending": Requirements incomplete or awaiting dean validation ← YELLOW BADGE
   - "cleared": ALL requirements validated by dean ← GREEN BADGE (only when 27/27)
   - "withholding": Has returned/rejected requirements ← RED BADGE

KEY PRINCIPLE:
--------------
Faculty clearance status is AUTOMATICALLY CALCULATED after dean validates requirements:

✓ Dean validates/checks each requirement (marks as validated or returned)
✓ Statistics show: "X Validated" (count of validated files)
✓ Faculty badge shows: "PENDING", "CLEARED", or "WITHHOLDING" (overall status)
   
   • If ANY requirements are returned → "WITHHOLDING" badge
   • If ALL requirements are validated → "CLEARED" badge (27/27 validated)
   • If requirements pending or not submitted → "PENDING" badge

WORKFLOW EXAMPLE:
-----------------
Faculty has 3 courses = 27 total requirements

Starting State:
  Statistics: 0 Validated, 1 Pending, 0 Returned, 26 Not Submitted
  Faculty Status Badge: PENDING (yellow)

After dean validates 1 file:
  Statistics: 1 Validated, 0 Pending, 0 Returned, 26 Not Submitted
  Faculty Status Badge: PENDING (yellow) ← Still pending, not all complete

After dean validates 10 more files:
  Statistics: 11 Validated, 0 Pending, 0 Returned, 16 Not Submitted
  Faculty Status Badge: PENDING (yellow) ← Still pending

After dean validates all 27 files:
  Statistics: 27 Validated, 0 Pending, 0 Returned, 0 Not Submitted
  Faculty Status Badge: CLEARED (green) ← Now shows cleared!

If dean returns 1 file:
  Statistics: 26 Validated, 0 Pending, 1 Returned, 0 Not Submitted
  Faculty Status Badge: WITHHOLDING (red) ← Changed to withholding

WHAT CHANGED:

   - "cleared" status is AUTO-SET when all requirements are validated by dean
   - "withholding" status is AUTO-SET when any requirements are returned
   - "pending" status when requirements are incomplete or awaiting validation

2. Database Migration (2026-03-06-rename-cleared-to-validated.sql):
   - requirement_submissions.status enum changed from "cleared" to "validated"

3. After each dean validation action (validate/return):
   - System automatically recalculates faculty clearance status

4. Manual override available (setFacultyClearanceStatus):
   - Dean can manually override status if needed for special cases
   - Normally not needed since system auto-calculates

================================================================================
PERIOD-SPECIFIC CLEARANCE STATUS (March 6, 2026 Update)
================================================================================

NEW REQUIREMENT:
----------------
Faculty clearance status must be specific to an academic year and semester.

REASON:
-------
- Faculty may be cleared for one semester but not for another
- Mixing clearance data from multiple periods (when filtering "All Years" or 
  "All Semesters") creates confusion and inaccuracy
- Clearance status should only be shown when viewing a specific period

IMPLEMENTATION:
---------------

1. UI Changes (requirements-monitoring.html):
   - Clearance status badge only shows when BOTH filters are set to specific values
   - When "All Years" OR "All Semesters" is selected:
     * Badge shows "N/A - Select specific year & semester"
     * All clearance action buttons are disabled
     * Warning message displayed explaining the requirement

2. Component Logic (requirements-monitoring.ts):
   - Added isSpecificPeriodSelected() method:
     * Returns true only when selectedAcademicYear !== 0 AND selectedSemester !== ''
   - All clearance methods (set/calculate) now validate specific period is selected
   - Shows warning dialog if methods are called without specific period

3. Backend Validation (dean-requirement.controller.js):
   - setFacultyClearanceStatus: Requires academic_year_id and semester parameters
     * Returns 400 error if either is missing
     * Error message: "Academic year and semester are required..."
   - updateFacultyClearanceStatus: Same validation added
   - Response messages now include period info for confirmation

4. API Parameters:
   - Frontend already passes academic_year_id and semester to backend
   - These parameters are now REQUIRED (not optional)
   - Backend validates presence before processing

HOW IT WORKS:
-------------

1. Dean selects faculty from dropdown
2. Dean MUST select specific academic year (e.g., "2026-2027")
3. Dean MUST select specific semester (e.g., "1st Sem")
4. Only then:
   ✓ Clearance status badge displays (CLEARED/WITHHOLDING/PENDING)
   ✓ Clearance action buttons become enabled
   ✓ Dean can validate requirements and manage clearance status

5. Cannot set clearance status when:
   ✗ "All Years" is selected
   ✗ "All Semesters" is selected
   ✗ Combination of "All" for either filter

EXAMPLE SCENARIOS:
------------------

Scenario A - Valid (Specific Period):
  Filters: Academic Year = "2026-2027", Semester = "1st Sem"
  Result: ✓ Clearance badge shows "CLEARED" (green)
          ✓ Action buttons enabled
          ✓ Can validate requirements and set clearance status

Scenario B - Invalid (All Years):
  Filters: Academic Year = "All Years", Semester = "1st Sem"
  Result: ✗ Badge shows "N/A - Select specific year & semester"
          ✗ Action buttons disabled with opacity
          ✗ Warning banner: "Please select a specific academic year..."

Scenario C - Invalid (All Semesters):
  Filters: Academic Year = "2026-2027", Semester = "All Semesters"
  Result: ✗ Badge shows "N/A - Select specific year & semester"
          ✗ Action buttons disabled
          ✗ Cannot manage clearance status

Scenario D - Invalid (All for Both):
  Filters: Academic Year = "All Years", Semester = "All Semesters"
  Result: ✗ Badge shows "N/A"
          ✗ All clearance features disabled
          ✗ Shows all faculty requirements across all periods (viewing only)

USER EXPERIENCE:
----------------
- Clear visual feedback when filters are set to "All"
- Disabled buttons prevent accidental actions
- Warning messages explain why features are unavailable
- Backend validation provides additional safety layer

LIMITATIONS:
------------
⚠️ KNOWN LIMITATION: Database Structure
- Faculty table stores single clearance_status, clearance_remarks, clearance_date
- These fields are NOT period-specific in the database schema
- Current implementation updates the faculty record but uses period filters
- For true period-specific storage, would need:
  * New table: faculty_clearance (faculty_id, academic_year_id, semester, status, date)
  * Migration to move existing data
  * Updated queries to fetch period-specific clearance

Current Workaround:
- Frontend enforces period-specific selection
- Backend validates period parameters are provided
- Last-set clearance status overwrites previous (no history stored per period)
- This is acceptable for current use case where dean focuses on active semester
